## Problem Statement

You are responsible for verifying a newly taped-out RV32IMC processor against the RISC-V architectural compliance test suite. The processor implements RV32IMC and must pass the `riscv-arch-test` suite before software is enabled.

**Setup:**
- Processor: RV32IMC
- Simulator: Spike (as golden reference)
- RTL simulation: Verilator-based testbench
- Compliance framework: riscv-arch-test v3.x (sail-based reference model)

**Tasks:**

1. Explain the structure of a RISC-V compliance test: what is a "test signature," how is it generated, and how is it compared against a reference?
2. The compliance test for `ADD` (`rv32i/add-01.S`) fails. The failure shows that test case 47 produces an incorrect result in the RTL simulation. Describe the complete diagnostic procedure to identify the root cause.
3. Write a minimal RISC-V compliance-style test for the `SLT` (Set Less Than) instruction in RISC-V assembly, including the test harness header, three test vectors, and the result capture mechanism.
4. The compliance test for `LW` passes on Spike but fails on the RTL simulation with a mismatched signature on the first test vector. The test vector is `LW a0, 0(s0)` where `s0` points to `0x80001000`. Describe three possible RTL-level root causes.
5. Describe how to integrate the compliance test suite into a Verilator-based regression workflow with automatic pass/fail detection.

## Solution

### Part 1: Compliance Test Structure — Signatures and Comparison

**Test structure overview:**
```
A RISC-V compliance test is an assembly program with three sections:

1. Test setup (test harness):
   - Initialise processor state: set up stack, configure CSRs if needed.
   - Jump to the first test case.

2. Test body (repeated for each test vector):
   - Load input values into registers.
   - Execute the instruction under test.
   - Store the result to the "signature" memory region.

3. Test termination:
   - Write the signature region bounds to tohost/fromhost.
   - Halt via HTIF (write 1 to tohost).
```

**The signature:**
The signature is a contiguous memory region (typically in a `.data.sig` section) where the test program writes its results. Each result is stored as a 32-bit word (or 64-bit on RV64). After the test completes, the simulator dumps the signature region to a file.

```
Signature file format (hex dump, one 32-bit word per line):
  deadbeef
  00000001
  7fffffff
  00000000
  ...

Reference signature: generated by running the test on the Sail golden model
                     (or Spike, for an initial baseline).
Implementation signature: generated by running on the RTL simulation.
Comparison: diff the two files; any mismatch = compliance failure.
```

**Why signatures rather than pass/fail flags:**
A signature captures the exact output values for all test vectors in one flat binary representation. This allows fine-grained comparison: if vector 47 fails, the diff shows exactly which line (word) in the signature file is wrong, narrowing the search space immediately.

**Signature capture mechanism (assembly):**
```asm
# After each instruction under test:
    add  a0, a1, a2        # instruction under test
    sw   a0, 0(t2)         # store result to signature region
    addi t2, t2, 4         # advance signature pointer

# At the end of the test:
    la   t1, sig_begin
    la   t2, sig_end
    RVTEST_IO_WRITE_STR(x31, "Test end.\n")
    # signal completion via HTIF
```

### Part 2: Diagnosing a Failing ADD Test Case

**Test case 47 context:**
The `rv32i/add-01.S` test has ~100+ test vectors covering: typical values, zero operands, overflow, large positive/negative numbers. Test case 47 is a specific (input1, input2, expected_output) triple.

**Step 1: Identify the exact failing inputs and expected/actual output**

```bash
# Extract test case 47 from the signature diff
diff reference_add.sig rtl_add.sig
# Output: line 47: expected 0x7FFFFFFF, got 0x80000000

# Trace back: what inputs produce the expected output 0x7FFFFFFF via ADD?
# Look at the test source to find test case 47:
grep -n "TEST_RR_OP" rv32i/add-01.S | head -50 | tail -5
# Or count the TEST_RR_OP macros to find the 47th one
```

**Step 2: Examine the failing test vector source**
```asm
# Suppose the 47th vector is:
TEST_RR_OP(47, add, 0x7FFFFFFF, 0x3FFFFFFF, 0x40000000)
# This means: 0x3FFFFFFF + 0x40000000 = 0x7FFFFFFF
# The RTL produced 0x80000000 instead.
```

**Step 3: Identify the specific computation**
```
0x3FFFFFFF + 0x40000000:
  Binary:
    0011_1111_1111_1111_1111_1111_1111_1111
  + 0100_0000_0000_0000_0000_0000_0000_0000
  = 0111_1111_1111_1111_1111_1111_1111_1111
  = 0x7FFFFFFF  (correct)

RTL produced: 0x80000000 = 0b1000_0000_0000_0000_0000_0000_0000_0000
  This is exactly 1 too large: an off-by-one or a carry propagation error.
```

**Step 4: Root cause hypotheses**

1. **Carry propagation bug in the ALU:** A specific carry chain in the 32-bit adder has a stuck-at-1 fault. The carry from bit 29 to bit 30 or bit 30 to bit 31 is incorrect, causing the result to be incremented by 1.

2. **Incorrect bit width in the ALU RTL:** The adder is implemented as 33-bit (to capture carry-out) and the result assignment incorrectly includes bit 32 (the carry-out), which for this input would be 0, but a wiring error connects the wrong bit.

3. **Two's complement overflow detection interfering with the result:** If the ALU has overflow detection logic and the detection logic incorrectly concludes overflow (wrongly computing the sign extension), it might saturate or modify the result. For `0x3FFFFFFF + 0x40000000 = 0x7FFFFFFF`, there is no signed overflow (two positive numbers with a positive result), but if the overflow check logic has an off-by-one, it might set the result to INT_MAX+1 = 0x80000000.

**Step 5: Reproduce in simulation with targeted waveform capture**

```bash
# Run the RTL simulation with the specific test
make sim TESTCASE=add-01 DUMP_WAVES=1

# Open the waveform in GTKWave
gtkwave sim.fst

# Add signals to the waveform:
#   dut.alu_a          -> check: should be 0x3FFFFFFF
#   dut.alu_b          -> check: should be 0x40000000
#   dut.alu_result     -> check: RTL output
#   dut.alu_carry_out  -> examine carry propagation
#   dut.adder_internal[32:0] -> if visible, inspect full adder output
```

**Step 6: Verify the fix**

After identifying the RTL bug (e.g., a missing carry-chain connection in the adder), fix the RTL, re-run the full `rv32i` compliance suite, and verify all tests pass.

### Part 3: Writing a Minimal SLT Compliance Test

```asm
# SLT compliance test (minimal, illustrative)
# Tests: SLT rd, rs1, rs2  (rd = 1 if rs1 < rs2 (signed), else 0)
#
# riscv-arch-test compatible structure

#include "model_test.h"
#include "arch_test.h"

RVTEST_ISA("RV32I")

.section .text.init
.globl _start
_start:
RVTEST_CODE_BEGIN

    # ---- Test vector 1: positive < positive (true case) ----
    # Inputs: rs1 = 5, rs2 = 10
    # Expected: rd = 1 (5 < 10 is TRUE)
    TEST_RR_OP_SLT(1, slt, 1, 5, 10)

    # ---- Test vector 2: positive > positive (false case) ----
    # Inputs: rs1 = 10, rs2 = 5
    # Expected: rd = 0 (10 < 5 is FALSE)
    TEST_RR_OP_SLT(2, slt, 0, 10, 5)

    # ---- Test vector 3: negative < positive (signed comparison) ----
    # Inputs: rs1 = -1 (0xFFFFFFFF), rs2 = 1
    # Expected: rd = 1 (-1 < 1 is TRUE in signed arithmetic)
    # (This distinguishes SLT from SLTU: SLTU would return 0 for 0xFFFFFFFF < 1)
    TEST_RR_OP_SLT(3, slt, 1, -1, 1)

RVTEST_CODE_END

.section .data.sig
.align 4
.globl begin_signature
begin_signature:
    .fill 3, 4, 0xdeadbeef   # 3 test results, each 4 bytes
.globl end_signature
end_signature:
```

**Macro definition for SLT (expanding the `TEST_RR_OP_SLT` macro):**
```asm
# TEST_RR_OP_SLT testnum, inst, result, val1, val2
# Loads val1->a0, val2->a1, executes inst a2, a0, a1, stores result to signature
.macro TEST_RR_OP_SLT testnum, inst, result, val1, val2
    li   a0, \val1
    li   a1, \val2
    \inst a2, a0, a1          # execute SLT: a2 = (a0 < a1) ? 1 : 0
    sw   a2, sig_offset(\testnum), t3  # store to signature
                                        # (t3 = base of signature region)
.endm
```

**In a complete riscv-arch-test format, `TEST_RR_OP` is a standard macro that:**
1. Loads the input values into registers.
2. Executes the instruction.
3. Writes the result to the next signature slot.
4. The signature region is a flat array of 32-bit words.

**Expected signature for these 3 test vectors:**
```
# reference_slt.sig:
00000001    # test 1: 5 < 10 = TRUE = 1
00000000    # test 2: 10 < 5 = FALSE = 0
00000001    # test 3: -1 < 1 = TRUE = 1 (signed comparison)
```

**Why test 3 is critical:** It distinguishes `SLT` (signed less than) from `SLTU` (unsigned less than). For `SLTU`, -1 (0xFFFFFFFF) would be treated as the largest unsigned value (4294967295), which is NOT less than 1, giving result = 0. If the RTL incorrectly implements `SLT` as `SLTU`, test 3 catches it.

### Part 4: Diagnosing LW Compliance Failure on RTL

The failing test: `LW a0, 0(s0)` where `s0 = 0x80001000`. Signature mismatch on first vector. Spike passes; RTL fails.

**Root cause 1: Memory address decoder gap**

The RTL SoC has a memory map decoder that routes addresses to different subsystems (SRAM, ROM, peripherals). The address `0x80001000` must map to the SRAM bank where the test data was loaded.

```
Diagnosis:
  Examine the SoC address decoder in the RTL:
    - Does address 0x80001000 fall within the SRAM range?
    - Common issue: SRAM mapped to 0x80000000-0x8000FFFF (64 KB);
      0x80001000 is within range but perhaps there is an off-by-one in the range check.
    - Check: grep -n "0x8000" rtl/addr_decoder.sv
    - Simulate with wave capture:
        data_addr == 0x80001000 -> mem_select should assert; check it does.
```

**Root cause 2: Load data alignment / byte enable error**

`LW` requires a 4-byte aligned address. `0x80001000` is word-aligned (divisible by 4), so misalignment is not the issue for this specific address. However, the byte enable logic in the data cache or memory wrapper may incorrectly gate the data:

```
Diagnosis:
  In the RTL memory interface, `LW` should assert all 4 byte enables: be = 4'b1111.
  If be = 4'b0001 (only the LSB) or be = 4'b0000 (no bytes enabled), the
  read data will be zero or the LSB only.
  
  Waveform check:
    - Inspect d_be (data byte enable) signal during LW: should be 4'b1111.
    - Inspect d_rdata (read data): should be the 32-bit value at 0x80001000.
    - If d_rdata is 0 or only the lower byte: byte enable bug.
```

**Root cause 3: Load data sign extension or selection**

For `LW` (Load Word), the full 32-bit word is loaded and sign-extended to the register width (trivial for RV32: no sign extension needed, the 32-bit word is the result). However, for `LH`/`LB`, sign extension is required. A common RTL bug is that the `LW` case incorrectly routes through a sign-extension path (designed for LH or LB) that truncates or sign-extends the 32-bit value incorrectly.

```
Diagnosis:
  Inspect the load result MUX in the EX/MEM or MEM/WB stage:
    - For LW: should select full 32-bit d_rdata directly.
    - For LH: should select bits [15:0] and sign-extend.
    - For LB: should select bits [7:0] and sign-extend.
  
  If the LW path erroneously selects [15:0] and sign-extends,
  the upper 16 bits of the loaded word will be lost.
  
  RTL check:
    - Find the load_result MUX: grep -n "mem_to_reg" rtl/pipeline_wb.sv
    - Check that funct3 = 3'b010 (LW) selects the full d_rdata[31:0].
    - Simulate with d_rdata = 0xDEADBEEF; if result = 0xFFFFBEEF, the bug
      is sign-extension treating the word as a halfword.
```

**Verilator testbench integration:**

```cpp
// sim/tb_top.cpp (Verilator C++ testbench)
#include "Vtop.h"     // Verilator-generated model
#include <cstdio>

int main(int argc, char **argv) {
    Verilated::commandArgs(argc, argv);
    Vtop *dut = new Vtop;

    // Load ELF test binary into simulated memory
    // (custom ELF loader or using spike's libriscv)
    load_elf(argv[1]);  // argv[1] = path to compliance test ELF

    // Run simulation until tohost != 0 (test completion signal)
    uint64_t max_cycles = 1000000;
    while (!Verilated::gotFinish() && max_cycles-- > 0) {
        dut->clk = 0; dut->eval();
        dut->clk = 1; dut->eval();

        // Check tohost memory-mapped register
        if (read_mem(TOHOST_ADDR) != 0) {
            uint64_t tohost_val = read_mem(TOHOST_ADDR);
            if (tohost_val == 1) {
                // Normal exit: dump signature
                dump_signature("rtl_output.sig", SIG_BEGIN_ADDR, SIG_END_ADDR);
                break;
            } else {
                // Non-zero exit: test explicitly failed
                fprintf(stderr, "Test signalled failure: tohost=0x%lx\n", tohost_val);
                exit(1);
            }
        }
    }

    delete dut;
    return 0;
}
```

**Shell script for batch compliance runs:**

```bash
#!/bin/bash
# scripts/run_compliance.sh
# Runs all rv32i compliance tests and compares signatures

ARCH_TEST_DIR=tests/compliance/riscv-arch-test
SAIL_REF=tests/compliance/sail-riscv
SIM_BIN=sim/obj_dir/Vtop
PASS=0
FAIL=0
ERRORS=()

for test_src in ${ARCH_TEST_DIR}/riscv-test-suite/rv32i_m/I/*.S; do
    test_name=$(basename ${test_src} .S)

    # Assemble and link the test
    riscv32-unknown-elf-gcc \
        -march=rv32i -mabi=ilp32 \
        -T ${ARCH_TEST_DIR}/riscv-test-suite/env/arch_test.ld \
        -I ${ARCH_TEST_DIR}/riscv-test-suite/env \
        -nostdlib -static \
        -o /tmp/${test_name}.elf \
        ${test_src}

    # Generate reference signature from Sail model
    sail_riscv_sim -e /tmp/${test_name}.elf > /tmp/${test_name}.ref.sig

    # Run on RTL simulation
    ${SIM_BIN} /tmp/${test_name}.elf
    # (testbench writes rtl_output.sig)
    cp rtl_output.sig /tmp/${test_name}.rtl.sig

    # Compare signatures
    if diff -q /tmp/${test_name}.ref.sig /tmp/${test_name}.rtl.sig > /dev/null 2>&1; then
        echo "PASS: ${test_name}"
        ((PASS++))
    else
        echo "FAIL: ${test_name}"
        diff /tmp/${test_name}.ref.sig /tmp/${test_name}.rtl.sig
        ERRORS+=("${test_name}")
        ((FAIL++))
    fi
done

echo ""
echo "Results: ${PASS} passed, ${FAIL} failed"
if [ ${FAIL} -gt 0 ]; then
    echo "Failed tests:"
    for err in "${ERRORS[@]}"; do echo "  - ${err}"; done
    exit 1
fi
exit 0
```

**Python signature comparison with detailed mismatch reporting:**

```python
#!/usr/bin/env python3
# scripts/compare_sigs.py

import sys

def compare_signatures(ref_file, rtl_file):
    with open(ref_file) as f:
        ref_lines = [l.strip() for l in f if l.strip()]
    with open(rtl_file) as f:
        rtl_lines = [l.strip() for l in f if l.strip()]

    if len(ref_lines) != len(rtl_lines):
        print(f"ERROR: Signature length mismatch: ref={len(ref_lines)}, rtl={len(rtl_lines)}")
        return False

    mismatches = []
    for i, (ref, rtl) in enumerate(zip(ref_lines, rtl_lines)):
        if ref != rtl:
            mismatches.append((i + 1, ref, rtl))

    if mismatches:
        print(f"FAIL: {len(mismatches)} mismatches:")
        for line_no, ref, rtl in mismatches:
            print(f"  Line {line_no:3d}: expected 0x{ref}, got 0x{rtl}")
        return False

    print(f"PASS: all {len(ref_lines)} signature words match")
    return True

if __name__ == "__main__":
    ok = compare_signatures(sys.argv[1], sys.argv[2])
    sys.exit(0 if ok else 1)
```

## Discussion

### Why the compliance test suite is not sufficient for full verification

Compliance tests verify that a processor produces architecturally correct outputs for a curated set of test vectors. They do not verify:

1. **Timing correctness:** A processor that takes 1000 cycles per instruction is compliant. CPI is not tested.
2. **Interrupt latency and handler correctness:** Many compliance tests run in M-mode with interrupts disabled.
3. **Multiprocessor consistency:** RISC-V memory model compliance (RVWMO) requires separate concurrency tests.
4. **Platform-specific peripherals:** The compliance suite tests the ISA, not SoC integration.
5. **Random instruction combinations:** The curated vectors miss instruction sequence interactions (forwarding bugs, pipeline hazards) that only appear with specific sequences. Co-simulation with random instruction generation (riscv-dv) fills this gap.

A complete verification strategy for a production RISC-V processor combines: compliance testing, co-simulation with random instruction streams, directed microarchitecture tests (forwarding, hazard, branch prediction), formal property checking of the pipeline, and application-level testing.

### The sail model vs. Spike as a reference

The riscv-arch-test suite has moved toward using the Sail formal model as the primary reference, rather than Spike:
- Sail is a formally specified language; the RISC-V Sail model is part of the official ISA specification.
- Sail-generated simulators are provably correct against the written specification.
- Spike is a well-maintained reference implementation but is not formally proven.

For most practical purposes, Spike and Sail agree. Discrepancies between them indicate either a Spike bug or a Sail model specification ambiguity — both of which are reported to the RISC-V community.
