# Problem 01: Trap Delegation

**Topic:** Privilege Architecture — `medeleg`, `mideleg`, trap routing  
**Difficulty:** Intermediate  
**Estimated time:** 25 minutes

---

## Problem Statement

You are writing the M-mode firmware initialisation code (similar to OpenSBI) for a new RISC-V SoC. The SoC supports M-mode and S-mode. It will run a Linux kernel in S-mode with user applications in U-mode.

The platform has the following interrupt sources:
- Machine timer interrupt (MTI): generated by hardware comparator `mtime >= mtimecmp`
- Machine software interrupt (MSI): used for inter-hart IPIs by the CLINT
- Supervisor software interrupt (SSI): written by S-mode software (used by Linux for SMP IPIs)
- Machine external interrupt (MEI): PLIC delivers external device interrupts to M-mode context
- Supervisor external interrupt (SEI): PLIC delivers external device interrupts to S-mode context

The following traps must be handled:

| Trap | Where handled | Reason |
|---|---|---|
| `ecall` from U-mode (cause 8) | S-mode (Linux syscall handler) | Normal system call path |
| `ecall` from S-mode (cause 9) | M-mode (SBI firmware) | Linux calls OpenSBI for timer, IPI, etc. |
| Instruction page fault (cause 12) | S-mode | Kernel handles demand paging |
| Load page fault (cause 13) | S-mode | Kernel handles demand paging |
| Store/AMO page fault (cause 15) | S-mode | Kernel handles demand paging |
| Illegal instruction (cause 2) | S-mode | Kernel handles emulation |
| Breakpoint (cause 3) | S-mode | Kernel handles ptrace / kgdb |
| Machine timer interrupt | M-mode | Firmware reprograms mtimecmp; injects STIP |
| Supervisor timer interrupt | S-mode (if delegated) | Kernel timer tick |
| Supervisor external interrupt | S-mode | Device driver interrupt handling |
| Machine external interrupt | M-mode | Firmware PLIC claim/complete |

**Part A:** Write the C or assembly code to initialise `medeleg` and `mideleg` for the above requirements. Justify every bit you set or leave clear.

**Part B:** After your delegation is configured, the Linux kernel is running in S-mode. A U-mode process executes `ecall`. Trace the complete hardware and firmware state changes from the `ecall` instruction to the moment the kernel's system call handler begins executing. State every CSR that is written and its new value (use symbolic names for runtime values).

**Part C:** The machine timer interrupt fires while Linux is executing U-mode code. Describe the complete sequence: M-mode handler entry, M-mode handler actions, and return to U-mode. Why must M-mode handle this interrupt rather than delegating it directly to S-mode?

---

## Solution

### Part A: medeleg and mideleg Configuration

**Exception delegation (medeleg):**

We need to delegate the following exception causes to S-mode:
- Cause 2 (Illegal instruction): bit 2
- Cause 3 (Breakpoint): bit 3
- Cause 8 (ecall from U-mode): bit 8
- Cause 12 (Instruction page fault): bit 12
- Cause 13 (Load page fault): bit 13
- Cause 15 (Store/AMO page fault): bit 15

Causes deliberately NOT delegated:
- Cause 9 (ecall from S-mode): stays in M-mode — this is the SBI call path
- Cause 11 (ecall from M-mode): never delegatable; M-mode handles its own ecalls
- Cause 0/4/6 (misaligned address): often kept in M-mode on platforms that emulate misaligned accesses in firmware; may be delegated if kernel handles them

```c
/* C equivalent — runs in M-mode firmware (e.g., OpenSBI platform_init) */

#define MEDELEG_ILLEGAL_INSN     (1UL << 2)
#define MEDELEG_BREAKPOINT       (1UL << 3)
#define MEDELEG_ECALL_U          (1UL << 8)
/* cause 9 (ecall from S-mode) deliberately omitted — stays in M-mode */
#define MEDELEG_INSN_PAGE_FAULT  (1UL << 12)
#define MEDELEG_LOAD_PAGE_FAULT  (1UL << 13)
/* cause 14 reserved */
#define MEDELEG_STORE_PAGE_FAULT (1UL << 15)

unsigned long medeleg_val =
    MEDELEG_ILLEGAL_INSN    |
    MEDELEG_BREAKPOINT      |
    MEDELEG_ECALL_U         |
    MEDELEG_INSN_PAGE_FAULT |
    MEDELEG_LOAD_PAGE_FAULT |
    MEDELEG_STORE_PAGE_FAULT;

csr_write(CSR_MEDELEG, medeleg_val);
/* medeleg = 0x0000B10C */
```

**Interrupt delegation (mideleg):**

```
mip/mie bit layout:
  Bit 1   SSIP  Supervisor software interrupt pending/enable
  Bit 3   MSIP  Machine software interrupt
  Bit 5   STIP  Supervisor timer interrupt
  Bit 7   MTIP  Machine timer interrupt
  Bit 9   SEIP  Supervisor external interrupt
  Bit 11  MEIP  Machine external interrupt
```

Interrupts to delegate to S-mode (S-mode handles them directly):
- SSIP (bit 1): supervisor software interrupt — S-mode programs this itself for SMP IPIs
- STIP (bit 5): supervisor timer interrupt — BUT see Part C; M-mode must intercept MTIP and then inject STIP, so STIP itself is delegated
- SEIP (bit 9): supervisor external interrupt — device IRQs via PLIC S-mode context

Interrupts NOT delegated (handled by M-mode firmware):
- MSIP (bit 3): machine software interrupt — firmware IPIs
- MTIP (bit 7): machine timer interrupt — `mtimecmp` is not accessible to S-mode; firmware must handle it
- MEIP (bit 11): machine external interrupt — PLIC M-mode context

```c
#define MIDELEG_SSI  (1UL << 1)
#define MIDELEG_STI  (1UL << 5)
#define MIDELEG_SEI  (1UL << 9)

unsigned long mideleg_val = MIDELEG_SSI | MIDELEG_STI | MIDELEG_SEI;
csr_write(CSR_MIDELEG, mideleg_val);
/* mideleg = 0x0000_0222 */
```

Assembly equivalent:
```asm
# Exception delegation
li    t0, 0x0000B10C    # bits: 2,3,8,12,13,15
csrw  medeleg, t0

# Interrupt delegation
li    t0, 0x00000222    # bits: 1,5,9 (SSI, STI, SEI)
csrw  mideleg, t0
```

### Part B: U-mode ecall Trace

Initial state: U-mode user process executing; `medeleg[8] = 1` (ecall from U delegated to S-mode).

**Step 1 — U-mode executes `ecall`:**

The `ecall` instruction generates exception cause 8 (ecall from U-mode). Because `medeleg[8] = 1` and the current privilege is U-mode (below M), the trap is taken to **S-mode**.

**Step 2 — Hardware writes S-mode trap CSRs atomically:**

```
scause  <- 8                        (ecall from U-mode)
sepc    <- PC_of_ecall              (address of the ecall instruction)
stval   <- 0                        (no relevant address for ecall)

sstatus.SPIE <- sstatus.SIE         (save current SIE)
sstatus.SIE  <- 0                   (disable S-mode interrupts)
sstatus.SPP  <- 0                   (previous privilege was U-mode = 0)

privilege    <- S-mode
PC           <- stvec               (kernel's trap entry vector)
```

M-mode CSRs (`mepc`, `mcause`, `mstatus`) are **not modified**.

**Step 3 — Kernel trap entry executes (S-mode, PC = stvec):**

The kernel saves all 32 registers plus `sepc` and `sstatus` to the current process's kernel stack frame (`struct pt_regs`):

```asm
# Simplified Linux RISC-V trap entry (arch/riscv/kernel/entry.S)
csrrw  sp, sscratch, sp     # swap sp with kernel stack pointer
sd     ra,  PT_RA(sp)
sd     gp,  PT_GP(sp)
# ... save all 32 registers ...
csrr   t0, sepc
sd     t0,  PT_EPC(sp)      # save sepc (ecall instruction address)
csrr   t0, sstatus
sd     t0,  PT_STATUS(sp)
```

**Step 4 — Dispatch to system call handler:**

```c
/* handle_exception() in kernel/traps.c */
if (scause == 8) {  /* ecall from U-mode */
    /* syscall number is in a7 (saved in pt_regs) */
    long syscall_nr = regs->a7;
    /* advance return address past ecall */
    regs->epc += 4;
    /* dispatch */
    regs->a0 = sys_call_table[syscall_nr](regs->a0, regs->a1, ...);
}
```

At this point the kernel's system call handler is executing in S-mode. `sepc` in memory (pt_regs) now points to the instruction after the `ecall`.

### Part C: Machine Timer Interrupt in U-mode

The machine timer interrupt (MTIP) fires while U-mode code is executing. `mideleg[7] = 0` — MTI is NOT delegated to S-mode.

**Step 1 — Timer fires, hardware traps to M-mode:**

Even though the current privilege is U-mode, MTIP goes to M-mode because it is not delegated:

```
mcause  <- 0x80000007             (interrupt bit set, code 7 = MTI)
mepc    <- PC_of_interrupted_umode_insn
mtval   <- 0

mstatus.MPIE <- mstatus.MIE       (save MIE, typically 1)
mstatus.MIE  <- 0                 (disable M-mode interrupts)
mstatus.MPP  <- 0 (U-mode)       (save current privilege)

privilege    <- M-mode
PC           <- mtvec             (M-mode firmware timer handler)
```

**Step 2 — M-mode firmware timer handler executes:**

```c
/* OpenSBI timer interrupt handler */
void sbi_timer_process(void)
{
    /* 1. Disable the machine timer interrupt to prevent re-entry */
    csr_clear(CSR_MIE, MIE_MTIE);

    /* 2. Inject a supervisor timer interrupt (STIP) so the kernel
          will handle the timer tick when we return to S-mode.
          STIP is delegated (mideleg[5]=1), so it will fire in S-mode. */
    csr_set(CSR_MIP, MIP_STIP);

    /* 3. Re-enable M-mode interrupts (not needed for simple timer handling,
          but required for nested interrupt support) */
    /* (optional) csr_set(CSR_MIE, MIE_MTIE); */
}
```

Why M-mode must handle MTIP: the `mtimecmp` register (the comparator that controls MTIP) is a **memory-mapped** register, not a CSR. S-mode code cannot directly write `mtimecmp` because:
- M-mode may protect `mtimecmp` with PMP (CLINT MMIO region protected from S/U).
- Even if S-mode could write `mtimecmp`, the SBI specification defines a clean abstraction (`sbi_set_timer`) so the OS is portable across different timer hardware.

The SBI timer flow:
1. Linux calls `sbi_set_timer(next_tick)` via `ecall` from S-mode.
2. M-mode SBI handler writes `mtimecmp = next_tick`.
3. When `mtime >= next_tick`: hardware sets MTIP.
4. M-mode timer handler: clear MTIP (by bumping `mtimecmp` to max), set STIP.
5. MRET returns to wherever was interrupted.
6. Because STIP is now pending and `mideleg[5]=1` and `sstatus.SIE=1`: supervisor timer interrupt fires immediately in S-mode.
7. Linux kernel timer tick runs; calls `sbi_set_timer` for next tick.

**Step 3 — MRET from M-mode back to U-mode:**

```
PC           <- mepc              (U-mode instruction that was interrupted)
privilege    <- mstatus.MPP = U   (return to U-mode)
mstatus.MIE <- mstatus.MPIE      (restore interrupt enable)
mstatus.MPIE <- 1
mstatus.MPP  <- U (reset)
```

**Step 4 — STIP fires immediately upon returning to U-mode:**

Since STIP is delegated and `sstatus.SIE = 1` (Linux has S-mode interrupts enabled), the pending STIP causes an immediate trap to S-mode before the U-mode instruction at `mepc` executes:

```
scause  <- 0x80000005             (interrupt, supervisor timer = 5)
sepc    <- mepc                   (same U-mode PC, now saved in sepc)
sstatus.SPP <- 0 (U-mode)
PC      <- stvec                  (Linux timer interrupt handler)
```

Linux timer handler: updates `jiffies`, runs `do_timer()`, potentially preempts the current process. Then `sret` returns to U-mode (or a different process if preempted).

## Discussion

### Key Insight: Why Not Delegate MTIP?

One might ask: why not set `mideleg[7] = 1` to let S-mode handle the machine timer directly? The answer is that `mtimecmp` is an M-mode resource (memory-mapped, often PMP-protected). If the kernel could directly receive MTIP, it would need to write `mtimecmp` to clear it — but it cannot access that register. The two-stage design (MTIP -> M-mode -> set STIP -> S-mode) decouples the hardware timer mechanism from the OS timer abstraction.

### Common Interview Mistakes

1. **Delegating ecall from S-mode (cause 9):** This would route SBI calls back to S-mode instead of M-mode firmware, breaking the OpenSBI interface. Cause 9 must stay in M-mode.

2. **Forgetting to advance `sepc` after ecall:** The hardware saves `sepc` pointing at the `ecall` instruction. If the handler does not add 4 before `sret`, execution returns to the `ecall` and infinite loops.

3. **Confusing MTIP and STIP:** MTIP (bit 7) is the hardware timer interrupt, requiring M-mode to reprogram `mtimecmp`. STIP (bit 5) is a software-injectable interrupt that M-mode sets in `mip` to notify S-mode. They are separate mechanisms.

4. **Assuming M-mode CSRs are written when a delegated trap fires:** When `medeleg[8]=1` and an ecall fires, only `sepc`, `scause`, `stval`, `sstatus` are written. `mepc` and `mcause` retain their previous values. The M-mode trap handler is not entered.

### Verification

After configuring `medeleg` and `mideleg`, firmware should verify the values read back correctly (WARL semantics may not support all bits on all implementations):

```asm
csrr  t0, medeleg
li    t1, 0x0000B10C
bne   t0, t1, delegation_config_error

csrr  t0, mideleg
li    t1, 0x00000222
bne   t0, t1, delegation_config_error
```

If a platform does not implement `medeleg` (M-mode only system), writing to it is silently ignored, and all traps go to M-mode — which is correct for a no-OS bare-metal system.
